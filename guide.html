How to Use This System

A complete operational guide — from daily ticket management to Notion sync, designer handoff, and multi-session context transfer.

Foundation

Golden Rules

Before everything else. These four rules make the system reliable.

RULE 01
HTML is the database
The JSON inside tickets.html is your canonical ticket store. Notion, spreadsheets, and chat messages are inputs — not sources of truth. The HTML wins.
RULE 02
Edit only at anchors
Every editable section has a CATALYST:ANCHOR comment. LLMs must target these and only these. Never modify raw HTML outside an anchor — it will break the renderer.
RULE 03
One file per domain
Ticket data in tickets.html. Design tokens in design-system.html. Specs in specs/. Never duplicate data across files — you'll end up with drift.
RULE 04
context.html first
Start every new session by sharing context.html. This gives any LLM the full project map without reading all files. Then share only the specific file needed for the task.

Daily Workflow

What a typical working day looks like with this system.

Morning: Review the dashboard

1
Open index.html
Check current ticket counts and recent activity. This tells you where things stand without reading every ticket.
2
Open tickets.html
Filter to your area of focus for the day. Use the area dropdown or URL param: tickets.html?area=backend
3
Pick your tickets
Move 1–3 tickets from Backlog to In Progress. Ask Claude to do this — paste the LLM ticket-update prompt below.

During the day: Capture work

1
New task comes in from Notion / Slack / meeting
Use the "Add ticket from Notion" or "Add new ticket" prompt. Claude formats it and updates tickets.html + index.html.
2
Ticket moves to review
Use "Update ticket status" prompt. Takes 10 seconds — just paste the ticket ID and new status.
3
Designer needs a spec
Open a new chat, share specs/_template.html + design-system.html. Claude builds the spec. You hand off the single HTML file.

End of day: Sync

1
Bring in any new Notion tasks
Copy tasks from Notion, paste into Claude with tickets.html attached, let Claude format and add them. Takes one chat turn.
2
Update index.html
Use the "Sync dashboard" prompt to recalculate stats and activity. Takes one chat turn.
Working with LLMs

LLM Operations

How any LLM — Claude, GPT, Gemini — can modify this system. The key is the CATALYST:LLM-OPERATIONS comment block inside tickets.html. Any LLM that reads the file reads those instructions automatically and knows what it can and cannot change.

💡 You never need to explain the system to a new LLM. Just paste context.html and the relevant file. The manifest comments do the briefing automatically.

What LLMs can do to this system

✓
Add new tickets
LLM appends a new JSON object to the ticket array in CATALYST:TICKETS:DATA and bumps the next ID counter in the manifest.
✓
Update ticket status, priority, or description
LLM finds the ticket by ID and edits the relevant field. Always updates the "updated" date.
✓
Sync stats to index.html
LLM updates the number values inside CATALYST:STATS and rewrites CATALYST:ACTIVITY.
✓
Create designer specs
LLM copies the template, fills all SPEC: sections, generates phone frame mockups in HTML/CSS, and registers the spec in specs/index.html.
✓
Extend the design system
LLM adds new components or tokens inside the appropriate CATALYST:DS:* section.
✗
LLMs must NOT touch
Any HTML outside anchor tags. CSS styles. JavaScript. The nav bar. Layout. These are read-only to LLMs.
⚠️ After any ticket edit: always ask the LLM to also sync index.html. The dashboard doesn't auto-update — the LLM needs to manually recount and update CATALYST:STATS and CATALYST:ACTIVITY.

Integrations

Notion → Catalyst

Claude handles this. Copy tasks from Notion, paste them into a chat with tickets.html attached, and Claude maps them to the ticket schema and updates the file. No scripts, no setup.

How to do it

1
In Notion, copy the tasks you want to bring across
Select rows in your database view, or copy the page content. Plain text is fine — Claude can read Notion's copy format.
2
Open a Claude chat and share tickets.html + index.html
Attach both files. Claude reads CATALYST:LLM-OPERATIONS automatically and knows the schema and rules.
3
Paste the "Add tickets from Notion" prompt, then paste your Notion content below it
The prompt is in the Chat Prompts section above — it tells Claude exactly how to map Notion fields to Catalyst fields and how to handle duplicates.
4
Claude returns updated file content — save it
Copy the output and overwrite tickets.html. If index.html was also updated, save that too.
💡 Tip: you don't need to copy every Notion task every time. Just copy the ones that changed or are new. Claude will check for duplicates by title and update existing tickets rather than creating doubles.

Status mapping Claude uses

Notion "Not started" → Catalyst "backlog" Notion "In progress" → Catalyst "in-progress" Notion "In review" → Catalyst "review" Notion "Done" → Catalyst "done"

If your Notion uses different status names, just tell Claude in the prompt: "My Notion calls it 'Active' — map that to 'in-progress'."

Building Out

Designer Handoff

The spec workflow — from ticket to a file your designer can open in any browser.

When to create a spec

Create a spec when a ticket moves from Backlog → In Progress and involves UI work. The spec replaces docx briefs, Figma links, and scattered Slack messages.

Full spec workflow

1
Open a new Claude chat
Separate from your PM chat — specs get their own focused session.
2
Share: specs/_template.html + design-system.html
These two files are all Claude needs. design-system.html provides the token reference; the template provides the structure.
3
Paste the "Create a designer spec" prompt
Fill in the ticket ID, screen names, and description. The more detail you give, the more accurate the spec.
4
Review the output
Claude returns a complete HTML file. Check: phone frame mockups render correctly, measurements match your intent, interaction descriptions are clear.
5
Save as specs/[name]-v1.html
Save the file in the specs/ folder. Update specs/index.html — use the "Register spec" prompt or manually add a row to CATALYST:SPECS:REGISTRY.
6
Send the single file to your designer
Email or message them the .html file. They open it in any browser. No Figma, no login, no external links.
7
Mark ticket as "review"
Update CAT-XXX status to "review" in tickets.html. The spec IS the review artifact.
💡 Versioning: when a spec needs significant changes, create v2.html rather than overwriting v1. Keep v1 in the registry marked "Deprecated" — you'll want the history later.

Context Transfer

How to start a new chat with the right context, without loading everything.

📌 The core principle: CATALYST:MANIFEST comments are at the top of every file. Any LLM reads them in seconds and understands the full picture without reading the file body. This is what makes targeted loading work.

Starting a new session (any LLM)

1
Share context.html only
This is the briefing doc. The LLM reads the CATALYST:MANIFEST comment at the top and knows every file, every anchor, and the current state.
2
State your task, then share one targeted file
"Backend sprint today — here's tickets.html." The LLM reads that file's manifest and targets only the backend ticket IDs.
3
Never share all files at once
You'll burn context tokens on irrelevant content. Rule: one task = one file (plus context.html).

Passing context between chats

1
Chat A does work, returns updated HTML
E.g. a spec session. Save the returned spec file.
2
Chat B picks up with updated files
The spec file now contains all the decisions Chat A made. Share it with Chat B. No summary needed — the file IS the context.
💡 HTML files as memory: because files are edited in place, each new session inherits the complete history of all previous sessions. The HTML isn't a report — it's a living record.

Ticket Schema

FieldType / ValuesNotes
id"CAT-001"3-digit zero-padded. Never change after creation. Check manifest for next available.
titlestring3–8 words. Imperative verb + subject. E.g. "Add push notification service."
descriptionstringOne sentence. What needs to be done, not why.
area"pitch-deck" | "ai-agents" | "mobile" | "backend"Lowercase hyphenated. Must match exactly — controls kanban column color.
status"backlog" | "in-progress" | "review" | "done"Flow is one direction: backlog → in-progress → review → done. Never set backward.
priority"low" | "medium" | "high" | "critical"Use critical sparingly. If everything is critical, nothing is.
created"YYYY-MM-DD"Set once on creation. Never change.
updated"YYYY-MM-DD"Update every time any field changes. Today's date.
